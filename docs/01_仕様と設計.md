# ML パイプライン設計書

## 概要

カリフォルニア住宅価格データを対象とした ML パイプラインのローカル開発・検証環境。
Docker Compose で完結する構成。メンバーへの MLOps 教育用途を主目的とする。

---

|---|---|---|
| データマート | PostgreSQL (docker-compose) | BigQuery |
| 推論モデル | LightGBM | LightGBM |
| 実験ログ | W&B (SaaS / offline) | W&B (SaaS) |

## パッケージ構成と責務

```
src/
├── ml/              ML コア
│   ├── pipeline/        データパイプライン + オーケストレーション
│   ├── trainer/         学習アルゴリズム (LightGBM)
│   └── evaluation/      精度評価 + 実験ログ (W&B)
├── api/             推論 API + Web UI (FastAPI + Jinja2)
└── share/           共通定義 + ロギング
```

### 責務分担

| パッケージ | やること | やらないこと |
|---|---|---|
| ml/pipeline | データ取得、前処理、特徴量エンジニアリング、学習フロー制御、seed、精度記録、エラーハンドリング | 学習アルゴリズム、評価計算 |
| ml/trainer | LightGBM 学習、モデル保存 | データ取得、W&B ログ |
| ml/evaluation | RMSE/R² 算出、metrics 保存、W&B 管理 | 学習、データ取得 |
| api | 推論エンドポイント、Web UI、lifespan でモデルロード | 学習、評価 |
| share | FEATURE_COLS/ENGINEERED_COLS/MODEL_COLS/TARGET_COL、BaseAppSettings、get_logger、Run ID 生成 | ビジネスロジック |

### 依存の方向

```
ml/pipeline → ml/trainer → ml/evaluation
ml/pipeline → share
ml/trainer  → share
api         → share
ml/evaluation は他パッケージに依存しない
share は他パッケージに依存しない
```

## データフロー

```
sklearn.datasets.fetch_california_housing
  ↓ seed.py
PostgreSQL (training_data / test_data)
  ↓ PostgresRepository.fetch_train() / fetch_test()
DataFrame
  ↓ preprocess: 欠損値補完 → 外れ値キャップ → 対数変換
  ↓ feature_engineering: BedroomRatio・RoomsPerPerson 生成
  ↓ LightGBM 学習 (固定ハイパーパラメータ, 10特徴量)
best model
  ↓ models/{run_id}/model.lgb に保存
  ↓ models/latest シンボリックリンク更新（アトミック）
  ↓ evaluation: RMSE/R² 算出 → metrics.json 保存 → W&B ログ
  ↓ PostgreSQL model_runs テーブルに run_id / rmse / r2 を記録
FastAPI lifespan で models/latest/model.lgb を app.state にロード
  ↓
POST /predict → preprocess_input (対数変換) → engineer_features_input (特徴量生成) → 推論 → JSON 結果
GET  /        → 予測フォーム (Web UI)
```

## モデルバージョニング

Run ID 形式: `YYYYMMDD_HHMMSS_{6桁UUID}`

```
models/
├── 20260416_222525_5bfd5f/
│   ├── model.lgb
│   └── metrics.json        {"rmse": 0.4653, "r2": 0.8392, "run_id": "..."}
├── 20260416_222540_05f983/
│   ├── model.lgb
│   └── metrics.json
└── latest -> 20260416_222540_05f983
```

- 学習を複数回実行するとモデルが蓄積され、metrics.json で比較可能
- API は `models/latest/model.lgb` を参照（再起動でモデル切り替え）
- symlink 更新はアトミック（`tmp.symlink_to()` → `tmp.replace()`）

## 前処理

`ml/pipeline/preprocess.py` で実装。学習・推論の両方で適用。

### 学習時 (`preprocess`)

| ステップ | 対象カラム | 内容 |
|---|---|---|
| 欠損値補完 | 全特徴量 + Price | 中央値で補完 |
| 外れ値キャップ | Population, AveOccup, AveRooms, AveBedrms | 99パーセンタイルで上限カット |
| 対数変換 | Population, AveOccup | `log1p()` で右裾を圧縮 |

### 推論時 (`preprocess_input`)

API の `/predict` で対数変換のみ適用（単一レコードのためキャップは不要）。

---

## 特徴量エンジニアリング

`ml/pipeline/feature_engineering.py` で実装。前処理の後、学習の前に適用。

### 学習時 (`engineer_features`)

| 生成特徴量 | 計算式 | 意味 |
|---|---|---|
| BedroomRatio | AveBedrms / AveRooms | 寝室が占める割合 |
| RoomsPerPerson | AveRooms / AveOccup | 1人あたりの部屋数 |

元の8特徴量 → 10特徴量に拡張。`share/schema.py` の `MODEL_COLS` で一元管理。

### 推論時 (`engineer_features_input`)

API の `/predict` で `preprocess_input` の後に適用。同じ2特徴量を生成。

---

## 精度記録

学習完了後、PostgreSQL の `model_runs` テーブルに精度を記録。

| カラム | 型 | 内容 |
|---|---|---|
| run_id | TEXT (PK) | Run ID |
| rmse | DOUBLE PRECISION | RMSE |
| r2 | DOUBLE PRECISION | R² |
| created_at | TIMESTAMPTZ | 記録日時 |

`repository.fetch_runs()` で新しい順に全 run を取得可能。

---

## 評価指標

numpy 自前実装 (scikit-learn 不使用):

- **RMSE**: `sqrt(mean((y_true - y_pred)²))`
- **R²**: `1 - SS_res / SS_tot` (定数ターゲット時はゼロ除算防止付き)

## 推論 API

FastAPI lifespan パターンでモデルを `app.state.booster` にロード（グローバル状態なし）。

| Method | Path | 説明 |
|---|---|---|
| GET | `/` | 予測フォーム (Jinja2) |
| POST | `/predict` | JSON 推論 |
| GET | `/health` | ヘルスチェック |

入力特徴量 (`share/schema.py` の `FEATURE_COLS` で一元管理):

| 特徴量 | 説明 |
|---|---|
| MedInc | 中央所得 |
| HouseAge | 築年数 |
| AveRooms | 平均部屋数 |
| AveBedrms | 平均寝室数 |
| Population | 人口 |
| AveOccup | 平均世帯人数 |
| Latitude | 緯度 |
| Longitude | 経度 |

内部で自動生成される特徴量 (`ENGINEERED_COLS`):

| 特徴量 | 計算式 | 説明 |
|---|---|---|
| BedroomRatio | AveBedrms / AveRooms | 寝室が占める割合 |
| RoomsPerPerson | AveRooms / AveOccup | 1人あたりの部屋数 |

モデルは `MODEL_COLS` (8 + 2 = 10特徴量) で学習・推論する。

> 価格の単位は $100,000。予測値 4.2103 = 約 $421,030。

## エラーハンドリング

`ml/pipeline/main.py` で各ステップを保護:

| ステップ | 失敗時の動作 |
|---|---|
| データ取得 | `logger.error` → 終了 |
| W&B 初期化 | `logger.warning` → 継続 |
| 学習 | `logger.error` → 終了 |
| W&B ログ送信 | `logger.warning` → 継続 |

W&B の障害で学習パイプライン全体が止まらない設計。

## 今後の追加対応検討項目

| # | 項目 | 目的 | 概要 |
|---|---|---|---|
| 1 | **データバリデーション** | データ起因のサイレント障害を防ぐ | 学習前にスキーマ検証 (カラム存在・型・値域) を実施。異常データを検知したら学習を中断し、明確なエラーを出す。Great Expectations 等の導入も検討 |
| 2 | **クオリティゲート** | 壊れたモデルのデプロイを防ぐ | 学習後に RMSE / R² が閾値を満たすかチェック。基準未達のモデルは `latest` symlink を更新しない。閾値は設定ファイルで管理 |
| 3 | **Champion/Challenger** | モデルの退行を防ぐ | 新モデル (Challenger) の精度を現行モデル (Champion) と比較し、改善した場合のみ `latest` を切り替える。`model_runs` テーブルの精度履歴を活用 |
| 4 | **特徴量重要度** | デバッグ・説明性 | LightGBM の `feature_importance()` を `metrics.json` に記録。どの特徴量が予測に寄与しているかを可視化し、モデルの説明性を確保 |
| 5 | **推論時バリデーション** | API 品質 | Pydantic `Field` で入力値の範囲制約 (緯度 32〜42、経度 -124〜-114 等) を追加。範囲外の入力に対して 422 エラーと明確なメッセージを返す |

### 依存関係

```
① データバリデーション → ② クオリティゲート → ③ Champion/Challenger
                                                      ↑
                                          model_runs テーブル (実装済み)

④ 特徴量重要度 ← 単独で追加可能
⑤ 推論時バリデーション ← 単独で追加可能
```

### 推奨実装順

1. **⑤ 推論時バリデーション** — 最小工数で API 品質が向上。Pydantic Field の追加のみ
2. **④ 特徴量重要度** — `train.py` に数行追加で実現。デバッグに即効性あり
3. **① データバリデーション** — 前処理の前にスキーマチェックを挟む
4. **② クオリティゲート** — ①の延長で閾値判定を追加
5. **③ Champion/Challenger** — ②の延長。`model_runs` テーブルとの比較ロジック
